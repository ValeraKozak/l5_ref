# Лабораторна робота 5

## Тема

Бізнес-логіка, сервіси, unit-тестування, рефакторинг.

## Реалізовані бізнес-сценарії

У проєкті реалізовано 4 базові сценарії бібліотечної системи:

1. `registerUser()` - реєстрація нового користувача.
Логіка: сервіс перевіряє обов'язкові поля, нормалізує email, не допускає дублювання користувачів і зберігає нового читача в репозиторії.

2. `findBooks()` - пошук книги за назвою або автором.
Логіка: сервіс перевіряє коректність пошукового рядка, а репозиторій виконує пошук без урахування регістру.

3. `issueBook()` - видача книги користувачу.
Логіка: сервіс перевіряє наявність книги та користувача, переконується, що книга ще не видана, і лише після цього змінює її стан.

4. `returnBook()` - повернення книги.
Логіка: сервіс перевіряє, що книга існує і дійсно видана, після чого знімає позначку видачі.

## Структура шарів

Проєкт організовано за шаблоном `Controller -> Service -> Repository`.

- `src/controllers/LibraryController.js` - точка входу, лише делегує виклики сервісу.
- `src/services/LibraryService.js` - містить усю бізнес-логіку.
- `src/repositories/BookRepository.js` і `src/repositories/UserRepository.js` - працюють з даними.
- `src/models/Book.js` і `src/models/User.js` - моделі сутностей.
- `tests/libraryService.test.js` - unit-тести.

## Як запустити

### 1. Запуск демонстраційного сценарію

```bash
npm start
```

### 2. Запуск unit-тестів

```bash
npm test
```

### 3. Перевірка стилю коду

```bash
npm run lint
```

## Рефакторинг і чистий код

Під час реалізації було усунуто типові запахи коду:

- Бізнес-логіка винесена з контролера в сервісний шар.
- Повторювані перевірки винесені в окремі допоміжні методи `ensureBookExists`, `ensureUserExists`, `validateRequiredString`.
- Методи мають одну відповідальність і зрозумілі назви.
- Пошук і нормалізація даних ізольовані в репозиторіях.

## Короткий звіт

Ця лабораторна демонструє побудову простої системи бібліотеки з чітким розділенням відповідальностей між шарами. Репозиторії інкапсулюють роботу з даними, сервіси реалізують правила предметної області, а контролер виступає тільки фасадом для виклику сценаріїв.

Unit-тести покривають як позитивні, так і негативні сценарії: успішну реєстрацію, пошук, видачу та повернення книги, а також помилки дублювання користувача, спробу видати вже видану книгу, спробу видати книгу неіснуючому користувачу та пошук із порожнім рядком. Така структура спрощує подальше розширення системи та робить код придатним до підтримки.
